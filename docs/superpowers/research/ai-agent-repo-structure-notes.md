## agents-md-and-claude-md
- AGENTS.md is a simple, open format for guiding coding agents, adopted by over 60,000 open-source projects. — https://agents.md/
- AGENTS.md complements README.md by containing the extra, sometimes detailed context coding agents need — build steps, tests, and conventions — that would clutter a human-facing README. — https://agents.md/
- AGENTS.md is just standard Markdown with no required fields or fixed headings; the agent simply parses whatever text is provided. — https://agents.md/
- AGENTS.md is stewarded by the Agentic AI Foundation under the Linux Foundation and is supported across many agents/tools, including those from OpenAI, Google, Cognition, UiPath, JetBrains, VS Code, Cursor, and GitHub Copilot. — https://agents.md/
- Claude Code reads CLAUDE.md, not AGENTS.md; if a repo already has an AGENTS.md, the recommended approach is to create a CLAUDE.md that imports it with `@AGENTS.md` syntax (optionally appending Claude-specific instructions below the import) so both tools read the same instructions without duplication. — https://docs.claude.com/en/docs/claude-code/memory
- A plain symlink (`ln -s AGENTS.md CLAUDE.md`) also works if no Claude-specific content is needed, though on Windows this requires Administrator privileges or Developer Mode, so the `@AGENTS.md` import is used instead. — https://docs.claude.com/en/docs/claude-code/memory
- Running `/init` in a repository that already has an AGENTS.md reads it and incorporates the relevant parts into the generated CLAUDE.md; `/init` also reads other tool configs such as `.cursorrules`, `.devin/rules/`, and `.windsurfrules`. — https://docs.claude.com/en/docs/claude-code/memory
- CLAUDE.md files are markdown files that give Claude persistent instructions for a project, a personal workflow, or an entire organization; Claude reads them at the start of every session. — https://docs.claude.com/en/docs/claude-code/memory

## progressive-disclosure
- Progressive disclosure means agents "incrementally discover relevant context through exploration," gradually assembling understanding layer by layer rather than loading all information upfront. — https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
- Under progressive disclosure, agents maintain lightweight identifiers (file paths, stored queries, web links) and use tools to dynamically load data into context at runtime — mirroring how humans rely on external organization systems rather than memorizing entire information corpuses. — https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
- Claude Code applies this strategy for data analysis: instead of loading full databases into context, the model writes targeted queries, stores results, and uses Bash commands like `head` and `tail` to analyze large data volumes without ever loading full data objects into context. — https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
- For Agent Skills, only the `name` and `description` metadata from all skills is pre-loaded into the system prompt at startup; Claude reads the full SKILL.md body only once the skill becomes relevant, and reads additional bundled files only as needed. — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/best-practices
- Skill authoring guidance recommends keeping SKILL.md as an overview that points to detailed materials (like a table of contents), keeping the SKILL.md body under 500 lines, splitting content into separate reference files as it grows, and keeping file references one level deep from SKILL.md so Claude reads complete files rather than partially previewing deeply nested references. — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/best-practices
- By contrast, CLAUDE.md files are described as being "naively dropped into context up front," while primitives like glob and grep allow Claude to navigate its environment and retrieve additional files just-in-time. — https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
- Claude Code best-practices guidance recommends using skills — rather than CLAUDE.md — for domain knowledge or workflows that are only relevant sometimes, since skills are loaded on demand without bloating every conversation. — https://www.anthropic.com/engineering/claude-code-best-practices
- Reference files longer than 100 lines should include a table of contents at the top so Claude can see the full scope of available information even when previewing with partial reads. — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/best-practices

## precedence-and-scope
- CLAUDE.md files can live at four scopes, listed broadest to most specific: managed policy (organization-wide, e.g. `/etc/claude-code/CLAUDE.md` on Linux/WSL), user instructions (`~/.claude/CLAUDE.md`, personal preferences across all projects), project instructions (`./CLAUDE.md` or `./.claude/CLAUDE.md`, team-shared via source control), and local instructions (`./CLAUDE.local.md`, personal and gitignored). — https://docs.claude.com/en/docs/claude-code/memory
- Claude Code reads CLAUDE.md files by walking up the directory tree from the working directory; discovered files are concatenated (not overriding one another) ordered from filesystem root down to the working directory, so instructions closer to where Claude was launched are read last, and within a directory `CLAUDE.local.md` is appended after `CLAUDE.md`. — https://docs.claude.com/en/docs/claude-code/memory
- CLAUDE.md and CLAUDE.local.md files in the directory hierarchy above the working directory load in full at launch, but files in subdirectories load on demand only when Claude reads files in those subdirectories. — https://docs.claude.com/en/docs/claude-code/memory
- For settings.json (as distinct from CLAUDE.md), Claude Code applies precedence highest to lowest as: Managed (cannot be overridden by anything) > command-line arguments (temporary session overrides) > Local (`.claude/settings.local.json`) > Project (`.claude/settings.json`) > User (`~/.claude/settings.json`, lowest, applies when nothing else specifies the setting). — https://docs.claude.com/en/docs/claude-code/settings
- Permission rules are a stated exception to override-based precedence: rather than one scope overriding another, permission rules merge across scopes. — https://docs.claude.com/en/docs/claude-code/settings
- A managed policy CLAUDE.md (deployed via MDM/Group Policy or a `claudeMd` key in `managed-settings.json`) applies to every session on the machine, loads before user and project CLAUDE.md, and cannot be excluded by individual settings, unlike ordinary ancestor CLAUDE.md files which can be skipped in monorepos via the `claudeMdExcludes` setting. — https://docs.claude.com/en/docs/claude-code/memory
- CLAUDE.md content is delivered as a user message after the system prompt (not as part of the system prompt itself), and both CLAUDE.md and auto memory are treated by Claude as context rather than enforced configuration — there is no guarantee of strict compliance, especially for vague or conflicting instructions across files. — https://docs.claude.com/en/docs/claude-code/memory
- For AGENTS.md, in nested/monorepo structures the closest AGENTS.md to the edited file wins, and explicit user chat prompts override everything, which enables each package in a monorepo to ship tailored instructions. — https://agents.md/

## what-is-a-skill
- A skill is "organized folders of instructions, scripts, and resources that agents can discover and load dynamically to perform better at specific tasks." — https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills
- "Skills extend Claude's capabilities by packaging your expertise into composable resources for Claude." — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- Skills are "reusable, filesystem-based resources that provide Claude with domain-specific expertise: workflows, context, and best practices that transform general-purpose agents into specialists." — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- Each Skill "packages instructions, metadata, and optional resources (scripts, templates) that Claude uses automatically when relevant." — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- Skills are "folders that include instructions, scripts, and resources that Claude can load when needed," and Claude "will only access a skill when it's relevant to the task at hand." — https://www.anthropic.com/news/skills
- Skills have four defining attributes: composable ("Skills stack together. Claude automatically identifies which skills are needed and coordinates their use"), portable ("Build once, use across Claude apps, Claude Code, and API"), efficient ("Only loads what's needed, when it's needed"), and powerful ("Skills can include executable code for tasks where traditional programming is more reliable than token generation"). — https://www.anthropic.com/news/skills
- In Claude Code, "Skills extend what Claude can do. Create a SKILL.md file with instructions, and Claude adds it to its toolkit. Claude uses skills when relevant, or you can invoke one directly with /skill-name." — https://docs.claude.com/en/docs/claude-code/skills
- Claude Code skills "follow the Agent Skills open standard, which works across multiple AI tools." — https://docs.claude.com/en/docs/claude-code/skills
- Guidance on when to create one: "Create a skill when you keep pasting the same instructions, checklist, or multi-step procedure into chat, or when a section of CLAUDE.md has grown into a procedure rather than a fact." — https://docs.claude.com/en/docs/claude-code/skills

## skill-file-format
- A SKILL.md file "must begin with YAML Frontmatter that contains a file name and description, which is loaded into its system prompt at startup." — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- Required frontmatter fields are `name` and `description`; `name` allows a maximum of 64 characters, only lowercase letters/numbers/hyphens, no XML tags, and cannot contain the reserved words "anthropic" or "claude". — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- `description` must be non-empty, maximum 1024 characters, contain no XML tags, and should state both what the skill does and when Claude should use it. — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- Progressive disclosure has three content levels with different load timing and token cost: Level 1 metadata (name/description, ~100 tokens, always loaded at startup into the system prompt); Level 2 the SKILL.md body (instructions/guidance, under 5k tokens, loaded only when the skill is triggered); Level 3+ bundled resources and scripts (effectively unlimited token cost, loaded/executed only as needed via bash). — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- "When Claude runs validate_form.py, the script's code never loads into the context window. Only the script's output... consumes tokens," making scripts more efficient than generated code. — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- Example bundle layout: `pdf-skill/` containing `SKILL.md` (main instructions), `FORMS.md` (form-filling guide), `REFERENCE.md` (detailed API reference), and `scripts/fill_form.py` (utility script), with SKILL.md referencing FORMS.md by name so it's "read only when filling out a form." — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- In Claude Code, "Every skill needs a SKILL.md file with two parts: YAML frontmatter between --- markers that tells Claude when to use the skill, and markdown content with the instructions Claude follows when the skill runs." — https://docs.claude.com/en/docs/claude-code/skills
- Claude Code's full frontmatter field set includes `name`, `description`, `when_to_use`, `argument-hint`, `arguments`, `disable-model-invocation`, `user-invocable`, `allowed-tools`, `disallowed-tools`, `model`, `effort`, `context`, `agent`, `hooks`, `paths`, and `shell` — all optional, though `description` is recommended. — https://docs.claude.com/en/docs/claude-code/skills
- Authoring guidance: "Keep SKILL.md under 500 lines. Move detailed reference material to separate files," and reference supporting files (e.g., `reference.md`, `examples.md`, `scripts/helper.py`) from SKILL.md so Claude knows what each contains and when to load it. — https://docs.claude.com/en/docs/claude-code/skills

## skill-placement
- Claude Code skill locations and scope: Enterprise (managed settings, "All users in your organization"), Personal (`~/.claude/skills/<skill-name>/SKILL.md`, "All your projects"), Project (`.claude/skills/<skill-name>/SKILL.md`, "This project only"), Plugin (`<plugin>/skills/<skill-name>/SKILL.md`, "Where plugin is enabled"). — https://docs.claude.com/en/docs/claude-code/skills
- "Project skills load from .claude/skills/ in your starting directory and in every parent directory up to the repository root, so starting Claude in a subdirectory still picks up skills defined at the root." — https://docs.claude.com/en/docs/claude-code/skills
- Skills also "load from nested .claude/skills/ directories below your working directory," e.g. skills in `packages/frontend/.claude/skills/` become available when editing files there, supporting monorepo setups; a nested skill sharing a name with another appears under a directory-qualified name like `apps/web:deploy`. — https://docs.claude.com/en/docs/claude-code/skills
- A `<skill-name>` entry in the enterprise, personal, or project locations can be a symlink to a directory elsewhere on disk; Claude Code follows the symlink, and if the same target is reachable from more than one location, it loads the skill once. — https://docs.claude.com/en/docs/claude-code/skills
- Claude Code watches skill directories for live changes: adding, editing, or removing a skill under `~/.claude/skills/`, project `.claude/skills/`, or a `.claude/skills/` inside an `--add-dir` directory "takes effect within the current session without restarting," but creating a brand-new top-level skills directory requires a restart. — https://docs.claude.com/en/docs/claude-code/skills
- The `--add-dir` flag/`/add-dir` command is an exception to its normal "grant file access only" behavior: "`.claude/skills/` within an added directory is loaded automatically," whereas `permissions.additionalDirectories` in settings.json grants file access only and does not load skills. — https://docs.claude.com/en/docs/claude-code/skills
- Sharing scope differs by surface: on claude.ai custom Skills are "Individual user only," on the Claude API they are "Workspace-wide," and in Claude Code they are "Personal (~/.claude/skills/) or project-based (.claude/skills/); can also be shared via Claude Code Plugins." — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview
- Custom Skills do not sync across surfaces: "Skills uploaded to claude.ai must be separately uploaded to the API... Claude Code Skills are filesystem-based and separate from both claude.ai and API." — https://docs.claude.com/en/docs/agents-and-tools/agent-skills/overview

## project-vs-personal-skills
- Personal skills are created under `~/.claude/skills/<skill-name>/SKILL.md`; the docs' getting-started example instructs `mkdir -p ~/.claude/skills/summarize-changes` and notes "Personal skills are available across all your projects." — https://docs.claude.com/en/docs/claude-code/skills
- Project skills are created under `.claude/skills/<skill-name>/SKILL.md` and apply "This project only." — https://docs.claude.com/en/docs/claude-code/skills
- Precedence when names collide across levels: "enterprise overrides personal, and personal overrides project." — https://docs.claude.com/en/docs/claude-code/skills
- "A skill at any of these levels also overrides a bundled skill with the same name. For example, a code-review skill in your project's .claude/skills/ replaces the bundled /code-review." — https://docs.claude.com/en/docs/claude-code/skills
- "Plugin skills use a plugin-name:skill-name namespace, so they cannot conflict with other levels." — https://docs.claude.com/en/docs/claude-code/skills
- If a skill and a command (from `.claude/commands/`) share the same name, "the skill takes precedence." — https://docs.claude.com/en/docs/claude-code/skills
- Project skills are meant to be shared with a team by committing them to version control ("Project skills: Commit .claude/skills/ to version control"), distinct from personal skills which live only in the individual user's home directory. — https://docs.claude.com/en/docs/claude-code/skills

## directory-layout
- AGENTS.md should be created at the root of the repository; for monorepos, place another AGENTS.md inside each package, and agents automatically read the nearest file in the directory tree, with the closest one taking precedence. — https://agents.md/
- Claude Code skills live in per-scope directory layouts: personal (`~/.claude/skills/<skill-name>/SKILL.md`), project (`.claude/skills/<skill-name>/SKILL.md`), plugin (`<plugin>/skills/<skill-name>/SKILL.md`), and enterprise (via managed settings). — https://docs.claude.com/en/docs/claude-code/slash-commands
- Skills also load from nested `.claude/skills/` directories below the working directory, so a monorepo package (e.g. `packages/frontend/.claude/skills/`) can provide its own skills that apply when working on that package. — https://docs.claude.com/en/docs/claude-code/slash-commands
- CLAUDE.md files can live at multiple scopes ordered from broadest to narrowest: managed policy (e.g. `/etc/claude-code/CLAUDE.md` on Linux), user (`~/.claude/CLAUDE.md`), project (`./CLAUDE.md` or `./.claude/CLAUDE.md`), and local (`./CLAUDE.local.md`). — https://docs.claude.com/en/docs/claude-code/memory
- For larger projects, instructions can be organized into a `.claude/rules/` directory with one topic per file (e.g. `testing.md`, `api-design.md`), and all `.md` files are discovered recursively including subdirectories like `frontend/` or `backend/`. — https://docs.claude.com/en/docs/claude-code/memory
- OKF bundles organize concepts as a directory hierarchy of markdown files, one file per concept, with optional `index.md` files for progressive disclosure and optional `log.md` files for chronological change history (e.g. `sales/datasets/index.md`, `sales/tables/orders.md`). — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/

## naming-for-discoverability
- Because agents "automatically read the nearest" AGENTS.md file in the directory tree, the placement and naming of the file (root vs. per-package) directly determines which instructions an agent applies to a given piece of work. — https://agents.md/
- In Claude Code, the directory name of a skill becomes the command typed to invoke it (e.g. `.claude/skills/deploy-staging/SKILL.md` becomes `/deploy-staging`), while the frontmatter `name` field only sets the display label shown in skill listings. — https://docs.claude.com/en/docs/claude-code/slash-commands
- When a nested skill shares a name with a root-level skill, the nested one is exposed under a directory-qualified name such as `apps/web:deploy` so both remain invocable without collision. — https://docs.claude.com/en/docs/claude-code/slash-commands
- Claude Code documentation instructs skill authors to "pay special attention to the `name` and `description`" of a skill, since "Claude will use these when deciding whether to trigger the skill," making those two fields the primary discoverability signal. — https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills
- In OKF, a concept's file path is its identity, and concepts cross-link to each other using ordinary markdown links (e.g. a column description linking to `[customers](/tables/customers.md)`), making the relationship graph discoverable purely through file/path naming and links. — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/
- Claude Code's `.claude/rules/` guidance recommends "a descriptive filename like `testing.md` or `api-design.md`" so a rule's topical scope is discoverable from its filename alone. — https://docs.claude.com/en/docs/claude-code/memory

## readme-for-agents

- AGENTS.md is meant to complement README files by holding "the extra, sometimes detailed context coding agents need: build steps, tests, and conventions" that would otherwise clutter a human-facing README. — https://agents.md/
- Suggested AGENTS.md sections include project overview, build and test commands, code style guidelines, testing instructions, security considerations, commit message/PR guidelines, and deployment steps. — https://agents.md/

## okf-bundles-in-repos
- OKF v0.1 represents a bundle of documents as "just files" that can be "shipped as a tarball or hosted in any git repo," with no complex compression scheme, no new runtime, and no required SDK. — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/
- Because OKF bundles are version controlled, they can live "in git alongside the code" they describe, keeping curated knowledge and code changes co-located and reviewable together. — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/
- An OKF bundle is "just markdown," so it is readable in any text editor, renders directly on GitHub, and is indexable by ordinary search tools without special tooling. — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/
- OKF's only required frontmatter field is `type`; the reserved-but-optional fields are `title`, `description`, `resource`, `tags`, and `timestamp`, keeping the format "minimally opinionated" for producers. — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/
- Google shipped reference implementations alongside the OKF spec: an enrichment agent that walks BigQuery datasets and drafts OKF documents, a static HTML visualizer that turns any OKF bundle into an interactive graph view, and three sample bundles (GA4 e-commerce, Stack Overflow, and Bitcoin public datasets). — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/
- Google Cloud's Knowledge Catalog was updated to ingest Open Knowledge Format bundles and serve that curated knowledge to agents. — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/

## docs-for-agents
- Claude Code's memory documentation notes CLAUDE.md content is "delivered as a user message after the system prompt, not as part of the system prompt itself," so specific, concise, well-structured writing directly affects how reliably an agent follows it. — https://docs.claude.com/en/docs/claude-code/memory
- Documentation guidance for CLAUDE.md recommends targeting "under 200 lines" per file, because "longer files consume more context and reduce adherence." — https://docs.claude.com/en/docs/claude-code/memory
- AGENTS.md guidance recommends keeping READMEs "concise and focused on human contributors" while moving agent-specific build/test/convention detail into the separate AGENTS.md file. — https://agents.md/
- A Claude Code skill's markdown body "loads only when it's used, so long reference material costs almost nothing until you need it"; the docs recommend keeping SKILL.md under 500 lines and moving detailed reference material into separate supporting files. — https://docs.claude.com/en/docs/claude-code/slash-commands
- Anthropic's Agent Skills architecture documents a three-level progressive disclosure pattern for agent-facing docs: skill metadata (name/description) loaded into the system prompt at startup, full SKILL.md content loaded only when relevant, and supplementary files loaded on demand. — https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills
- OKF documents are designed to serve both audiences from the same file: agents "parse the directory, read YAML frontmatter, follow markdown links, and update files," while humans "browse in any text editor, view on GitHub, or use static visualizers" — no separate translation layer is required. — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/

## machine-readable-metadata
- A Claude Code `SKILL.md` file "must begin with YAML frontmatter"; only `description` is recommended, while `name`, `disable-model-invocation`, `allowed-tools`, and other fields are optional structured metadata that Claude parses to decide when and how to use the skill. — https://docs.claude.com/en/docs/claude-code/slash-commands
- Claude Code project rules use YAML frontmatter with a `paths` field of glob patterns so a rule file is scoped and loaded only when Claude works with matching files, e.g. `paths: ["src/api/**/*.ts"]`. — https://docs.claude.com/en/docs/claude-code/memory
- OKF frontmatter is YAML with one required field (`type`) and several reserved fields (`title`, `description`, `resource`, `tags`, `timestamp`) that both agents and query tooling can read as structured metadata alongside the markdown body. — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/
- If a `SKILL.md` file's frontmatter YAML is malformed, "Claude Code loads the skill body with empty metadata," so the skill can still be invoked by name but Claude has no `description` field left to match against — showing the frontmatter is parsed as a distinct metadata block from the body. — https://docs.claude.com/en/docs/claude-code/slash-commands
- Anthropic's documentation states the agent "pre-loads the `name` and `description` of every installed skill into its system prompt" at startup, using YAML frontmatter as a lightweight discovery index before any body content is read. — https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills
- OKF's "minimally opinionated" design principle means the spec mandates only the `type` field in frontmatter while leaving all other structured metadata producer-defined, prioritizing a shared machine-readable contract over any specific tool or schema. — https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/

## mcp-and-tool-config
- MCP servers are configured in a `.mcp.json` file at the project root, using a `mcpServers` object; this file is designed to be checked into version control so all team members share the same tools. — https://code.claude.com/docs/en/mcp
- Claude Code supports three MCP installation scopes: local (default, stored in `~/.claude.json`, private to current project), project (stored in `.mcp.json`, shared via version control), and user (stored in `~/.claude.json`, private but available across all projects). — https://code.claude.com/docs/en/mcp
- `.mcp.json` supports environment variable expansion with `${VAR}` and `${VAR:-default}` syntax in the `command`, `args`, `env`, `url`, and `headers` fields; if a required variable has no default and isn't set, Claude Code fails to parse the config. — https://code.claude.com/docs/en/mcp
- A per-server tool execution timeout can be set with a `timeout` field in milliseconds in a `.mcp.json` entry (e.g. `"timeout": 600000` for ten minutes), overriding the `MCP_TOOL_TIMEOUT` environment variable for that server only. — https://code.claude.com/docs/en/mcp
- Claude Code displays a warning before connecting to any MCP server and advises users verify they trust it, since servers that fetch external content can expose users to prompt injection risk. — https://code.claude.com/docs/en/mcp
- For security, Claude Code prompts for approval before using project-scoped servers loaded from a committed `.mcp.json` file, and `claude mcp list`/`claude mcp get` show them as "Pending approval" until the workspace is trusted. — https://code.claude.com/docs/en/mcp

## slash-commands-and-hooks
- Custom slash commands are created as Markdown files in `.claude/commands/` (project scope) or `~/.claude/commands/` (personal scope); a file at `.claude/commands/deploy.md` creates the `/deploy` command, and these files still work even though "commands have been merged into skills" as the current recommended approach. — https://code.claude.com/docs/en/slash-commands
- Slash command/skill content supports argument substitution via `$ARGUMENTS` (all arguments as typed), `$ARGUMENTS[N]`/`$N` (individual positional arguments by 0-based index), and named `$name` substitutions declared in an `arguments` frontmatter list. — https://code.claude.com/docs/en/slash-commands
- Hooks are configured in JSON settings files (`~/.claude/settings.json`, `.claude/settings.json`, or `.claude/settings.local.json`) under a `hooks` key, nested by event name (e.g. `PreToolUse`), then by `matcher` (e.g. tool name pattern), then by an array of hook handler objects. — https://code.claude.com/docs/en/hooks
- For `PreToolUse` hooks, exit code `2` blocks the tool call and feeds stderr back to Claude, while other non-zero exit codes are non-blocking and just show stderr in the transcript; exit code `0` lets Claude Code parse stdout as JSON for fine-grained control (e.g. `permissionDecision`, `decision`, `additionalContext`). — https://code.claude.com/docs/en/hooks
- Hook `matcher` fields can be an exact tool name, a pipe-separated list (`Edit|Write`), `"*"`/empty/omitted to match everything, or an unanchored regex for other characters (e.g. `mcp__memory__.*` to match all tools from an MCP server named "memory"). — https://code.claude.com/docs/en/hooks
- Hooks and MCP servers both receive a `CLAUDE_PROJECT_DIR` environment variable pointing at the project root, letting hook scripts reference project-local paths (e.g. `${CLAUDE_PROJECT_DIR}/.claude/hooks/validate-rm.sh`) regardless of the current working directory. — https://code.claude.com/docs/en/hooks

## determinism-and-reproducibility
- Anthropic's guidance explicitly distinguishes the two mechanisms: "Hooks run scripts automatically at specific points in Claude's workflow. Unlike CLAUDE.md instructions which are advisory, hooks are deterministic and guarantee the action happens," recommending hooks for "actions that must happen every time with zero exceptions." — https://code.claude.com/docs/en/best-practices
- Best-practice guidance recommends checking `CLAUDE.md` into git so a team can contribute to and share the same persistent project context, stating "the file compounds in value over time." — https://code.claude.com/docs/en/best-practices
- For batch/scripted automation across many files, the `--allowedTools` CLI flag is used to scope exactly which tools a non-interactive `claude -p` invocation may use, which "matters when you're running unattended." — https://code.claude.com/docs/en/best-practices
- Non-interactive mode (`claude -p "prompt"`) supports `--output-format json` or `--output-format stream-json --verbose` specifically so results can be parsed programmatically in CI pipelines, pre-commit hooks, or scripts, rather than relying on free-text prose output. — https://code.claude.com/docs/en/best-practices
- Reproducible/verifiable agent behavior is driven by giving Claude "a check it can run" (tests, a build, a linter, a script that diffs output against a fixture) so the agent gets a pass/fail signal instead of relying on subjective judgement of when work "looks done." — https://code.claude.com/docs/en/best-practices
- `settings.json` files support a `$schema` field (e.g. `"$schema": "https://json.schemastore.org/claude-code-settings.json"`) for autocomplete and validation in editors like VS Code/Cursor, and most settings keys (including `permissions` and `hooks`) reload without a restart when the file changes, while `model` and `outputStyle` require a restart or `/model`/`/clear`. — https://code.claude.com/docs/en/settings.md

## guardrails-and-permissions
- The SDK/Claude Code evaluates tool permission in a fixed six-step order: hooks run first (can deny outright), then `deny` rules, then `ask` rules, then the active permission mode, then `allow` rules, and finally the `canUseTool` callback if nothing else resolved it. — https://code.claude.com/docs/en/agent-sdk/permissions
- The supported permission modes are `default` (no auto-approvals), `dontAsk` (denies anything not pre-approved instead of prompting), `acceptEdits` (auto-approves file edits and filesystem operations like `mkdir`/`rm`/`mv`), `bypassPermissions` (approves everything reaching that step), `plan` (file edits always route to `canUseTool`, never auto-approved), and `auto` (a model classifier approves/denies each call). — https://code.claude.com/docs/en/agent-sdk/permissions
- `allowed_tools` does not constrain `bypassPermissions` mode: setting `allowed_tools=["Read"]` alongside `permission_mode="bypassPermissions"` still approves every tool including `Bash`, `Write`, and `Edit`; to block specific tools under `bypassPermissions` you must use `disallowed_tools`, since deny rules are still enforced even in that mode. — https://code.claude.com/docs/en/agent-sdk/permissions
- Claude Code's OS-level sandboxing (Seatbelt on macOS, bubblewrap on Linux) restricts filesystem access to the current working directory and restricts network access to an allowlist enforced by a proxy outside the sandbox, and internal testing showed sandboxing produced an 84% reduction in permission prompts compared to the prior prompt-per-action model. — https://www.anthropic.com/engineering/claude-code-sandboxing
- Declarative permission rules are configured in `.claude/settings.json` (or `~/.claude/settings.json`, `.claude/settings.local.json`) as `allow`/`deny`/`ask` arrays using a `"Tool(pattern)"` syntax (e.g. `"Bash(npm run test *)"`, `"Read(./.env)"`), and rules merge across settings scopes rather than one scope overriding another; project-scope allow rules require workspace trust to take effect while local-scope rules (personal, gitignored) do not. — https://code.claude.com/docs/en/settings.md
- Anthropic's containment strategy for Claude Code is described as a "human-in-the-loop sandbox": credentials/egress controls plus filesystem isolation were adopted after an internal red-team incident in which a phished employee's prompt caused Claude to exfiltrate AWS credentials in 24 of 25 retries, and Anthropic's stated guiding principle is to "design for containment at the environment layer first, then steer behavior at the model layer." — https://www.anthropic.com/engineering/how-we-contain-claude
