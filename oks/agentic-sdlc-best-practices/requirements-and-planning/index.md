# Requirements And Planning

Once an agent is the one turning a plan into code, the work that happens before
implementation carries more weight than it did. The agent builds from what the
spec says, not from what you meant, so ambiguity that a human teammate would
quietly resolve instead gets implemented literally — which makes the written plan
the primary artifact and reviewing it the cheapest place to catch a wrong
direction. These concepts move from why the spec matters, to how to cut the work
into buildable pieces, to using the agent itself to sharpen the requirements.

## Concepts

- [specs as the primary artifact](/oks/agentic-sdlc-best-practices/requirements-and-planning/specs-as-the-primary-artifact.md) — when the agent is the builder, the spec is closer to the program; ambiguity costs multiply, and plan review beats diff review as a quality gate.
- [task decomposition](/oks/agentic-sdlc-best-practices/requirements-and-planning/task-decomposition.md) — cutting work into agent-sized, independently verifiable units, the orchestrator's core skill.
- [requirements elicitation with ai](/oks/agentic-sdlc-best-practices/requirements-and-planning/requirements-elicitation-with-ai.md) — using agents to draft, interrogate, and prototype requirements while the human keeps ownership of acceptance.
