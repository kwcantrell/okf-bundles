# Applications

This is where the rest of the bundle gets used. The foundations, context, harness,
and verification areas explain *why* and *how* agentic SDLC works; this area turns
that into procedures an agent can run. Every concept here contains an explicit,
reusable procedure — a numbered checklist or an annotated template/skeleton — not
just prose about the topic. These concepts carry the procedures for the bundle's
three consumer use cases: **an agent authoring an AGENTS.md**, **generating a
coding-agent system prompt**, and **designing repo tools and middleware** — with
two further concepts on packaging reusable skills and on staging adoption.

## Concepts

- [authoring an AGENTS.md](/oks/agentic-sdlc-best-practices/applications/authoring-an-agents-md.md) — a derive-from-evidence procedure and skeleton template for producing a repo-specific AGENTS.md that encodes what the repo actually does, defaulting to conservative autonomy.
- [generating system prompts](/oks/agentic-sdlc-best-practices/applications/generating-system-prompts.md) — compiling repo and SDLC knowledge into a coding-agent system prompt: gather, set altitude, encode gates and verification, add examples, then evaluate with evals rather than vibes.
- [designing repo tools and middleware](/oks/agentic-sdlc-best-practices/applications/designing-repo-tools-and-middleware.md) — mining SDLC pain points and deciding instruction vs. tool vs. middleware, then proving the change moved an eval (the +13.7-point harness case).
- [skills libraries](/oks/agentic-sdlc-best-practices/applications/skills-libraries.md) — packaging process knowledge as small, triggered, composable skills instead of one monolithic prompt, following the superpowers pattern.
- [adopting agentic SDLC](/oks/agentic-sdlc-best-practices/applications/adopting-agentic-sdlc.md) — staged adoption paths for individual developers, engineering leaders, and organizations, starting where verification is cheap and growing autonomy with demonstrated trust.
